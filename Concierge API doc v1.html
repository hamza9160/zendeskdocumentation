Partner Integration Guide

DocUpdate Concierge API

Concierge enables healthcare professionals to accelerate access to samples, patient support programs, education, reps, and more, all through a single integration with DocUpdate.

Version  1.0
Released  April 2026
Auth  API Key
01

Overview

The DocUpdate Concierge API enables healthcare professionals (HCPs) to submit requests directly from your application. DocUpdate acts as the middleware layer between your application and our underlying messaging engine — your integration never communicates directly with any third-party messaging provider.

Capabilities

CapabilityDescription
Request SubmissionSubmit sample requests, rep visit schedules, and custom requests
Request TrackingRetrieve active and archived requests for a given user
MessagingReal-time messaging via WebSocket on a specific request conversation
Contact PreferencesSet and update a user's preferred contact hours
User ManagementRegister and manage HCP user accounts

Architecture

Your application communicates exclusively with DocUpdate endpoints. DocUpdate handles all downstream processing.

Your Application
HTTPS + API Key  ·  WSS + JWT (messaging)
DocUpdate Concierge API
api.docupdate.io/concierge/v1
Internal only — not exposed to partners
Messaging Engine

Messaging UI Ownership

DocUpdate provides a WebSocket connection for real-time messaging. Your application is responsible for rendering the messaging interface. Messages are delivered with a role field (user or agent) that you can use to differentiate and style messages on each side of the conversation.

02

Authentication

All requests to the DocUpdate Concierge API are authenticated using an API key passed in the request header.

API Key Header

Header
X-DocUpdate-API-Key: your_api_key_here

Include this header on every request. Requests without a valid API key will receive a 401 Unauthorized response.

Rules

RuleDetail
TransportHTTPS only. HTTP requests will be rejected.
Key RotationContact your integration rep to rotate a compromised key.
EnvironmentsSeparate keys are issued for sandbox and production.
Obtaining an API Key: Contact your DocUpdate integration representative to receive your API key. Keep your key secure and do not expose it in client-side code.

WebSocket Authentication

The WebSocket connection used for real-time messaging (see Section 06 — Messaging) uses a separate authentication mechanism. Pass the following headers on the WebSocket upgrade request:

Headers
Authorization: Bearer {jwtToken}
npi:           1234567890
HeaderDescription
AuthorizationA JWT token identifying the HCP. Generated server-side from the user's NPI, name, and email. Expires after 6 hours — re-authenticate before opening a new connection.
npiThe HCP's National Provider Identifier. Used to associate the connection with the correct user account.
The X-DocUpdate-API-Key header is not used for WebSocket connections. The JWT replaces it for the messaging channel.

04

User Management

Before an HCP can submit requests or send messages, they must be registered with DocUpdate. Registration should occur automatically following user login in your application.

Register User

POST/users

Creates a new HCP user account. If the user already exists (identified by NPI), the existing account is returned.

Request Body

JSON
{
  "npi":       "1234567890",
  "email":     "dr.smith@practice.com",
  "firstName": "Jane",
  "lastName":  "Smith"
}
FieldTypeDescription
npirequired stringNational Provider Identifier. Used as the unique identifier for the user.
emailrequired stringUser's email address.
firstNamerequired stringUser's first name.
lastNamerequired stringUser's last name.
Phone number: The user's phone number is not stored at registration. It is collected at the point of request submission as part of the metadata object in POST /conversations (see Section 05), since contact preferences may vary per request.

Response — 201 Created / 200 OK

JSON
{
  "userId":    "usr_abc123",
  "npi":       "1234567890",
  "email":     "dr.smith@practice.com",
  "firstName": "Jane",
  "lastName":  "Smith",
  "createdAt": "2026-04-06T12:00:00Z"
}
User Registration Flow
flowchart TD A([App Login]) --> B["POST /users\nnpi · email · firstName · lastName"] B --> C{HTTP Response} C -- 201 Created --> D[New user registered] C -- 200 OK --> E[Existing user returned] D --> F([Store userId — ready to submit requests]) E --> F
05

Submitting a Request

Create Conversation

POST/conversations

Creates a new Concierge request conversation for the specified user.

Request Body

JSON
{
  "npi":         "1234567890",
  "requestType": "Request Free Samples",
  "message":     "I would like to request samples of Medication X.",
  "metadata": {
    "email": "dr.smith@practice.com",
    "name":  "Dr. Jane Smith",
    "phone": "404-555-0100"
  }
}
FieldTypeDescription
npirequired stringNPI of the submitting HCP.
requestTyperequired stringOne of: Request Free Samples, Schedule Rep Visit, Custom Request
messagerequired stringThe user's request message. Max 200 characters.
metadata.emailrequired stringUser's email address.
metadata.namerequired stringUser's full name.
metadata.phonerequired stringUser's phone number.

Response — 201 Created

JSON
{
  "conversationId": "conv_xyz789",
  "status":         "new",
  "requestType":    "Request Free Samples",
  "createdAt":      "2026-04-06T12:05:00Z"
}
Store the conversationId returned. It is required to retrieve messages and send follow-up messages on this request.
Submit Request Flow
flowchart TD A([User fills request form]) --> B["POST /conversations\nnpi · requestType · message · metadata"] B --> C{HTTP Response} C -- 201 Created --> D[Store conversationId] C -- 4xx / 5xx --> E([Show error to user]) D --> F([Show success — request submitted])
06

Messaging

Messaging uses a WebSocket connection instead of REST. A single persistent connection handles both receiving the message history and sending new messages in real time.

Connect

WSS/conversations/{conversationId}/messages

Opens a WebSocket connection for the specified conversation. Authentication is passed via the Authorization header on the upgrade request.

Connection Headers

Headers
Authorization: Bearer {jwtToken}
npi:           1234567890
On successful connection, the server immediately sends a history event containing all existing messages for the conversation in chronological order.

Server → Client Events

Event: history

Sent once on connect. Contains the full message history for the conversation.

JSON
{
  "type":           "history",
  "conversationId": "conv_xyz789",
  "messages": [
    {
      "messageId": "msg_001",
      "role":      "user",
      "text":      "I would like to request samples of Medication X.",
      "sentAt":    "2026-04-06T12:05:00Z"
    },
    {
      "messageId": "msg_002",
      "role":      "agent",
      "text":      "Thank you! We will process your request shortly.",
      "sentAt":    "2026-04-06T12:07:00Z"
    }
  ]
}

Event: new_message

Sent in real time whenever a new message is added to the conversation (from either party).

JSON
{
  "type":      "new_message",
  "messageId": "msg_003",
  "role":      "agent",
  "text":      "Your samples have been shipped.",
  "sentAt":    "2026-04-06T14:22:00Z"
}

Client → Server Events

Event: send_message

Sent by the client to post a new message on the conversation.

JSON
{
  "type": "send_message",
  "npi":  "1234567890",
  "text": "Any update on my sample request?"
}

The server responds with a new_message event broadcast to all connected clients (including the sender) to confirm the message was received.

UI Tip: Use the role field to position messages in your interface. Messages with role: "user" should appear on the right; role: "agent" on the left.
Messaging WebSocket Flow
flowchart TD A([Open conversation screen]) --> B["Connect\nWSS /conversations/{id}/messages"] B --> C{Connected?} C -- No --> D([Show error — retry]) C -- Yes --> E["Receive 'history' event"] E --> F[Render full message history] F --> G{Waiting for events} G -- Agent replies --> H["Receive 'new_message' event"] H --> I[Append message to UI] I --> G G -- User sends message --> J["Send 'send_message' event\ntype · npi · text"] J --> K["Server broadcasts 'new_message'"] K --> I
07

Request Management

Use these endpoints to retrieve a user's request history. Each item in the response includes a conversationId — this is the same ID used to open the WebSocket messaging connection for that request (see Section 06 — Messaging).

Get Active Requests

GET/users/{npi}/requests/active

Returns all active requests for the specified HCP. Active requests have a status of new, open, or pending.

Response — 200 OK

JSON
{
  "requests": [
    {
      "conversationId": "conv_xyz789",
      "requestType":    "Request Free Samples",
      "status":         "open",
      "createdAt":      "2026-04-06T12:05:00Z",
      "updatedAt":      "2026-04-06T14:00:00Z"
    }
  ]
}

Get Archived Requests

GET/users/{npi}/requests/archived

Returns all archived (non-active) requests.

Status Reference

StatusCategoryDescription
newActiveRequest received, awaiting initial response.
openActiveRequest is being actively worked on.
pendingActiveAwaiting a response from the HCP.
solvedArchivedRequest was completed successfully.
closedArchivedRequest was resolved and closed.
holdArchivedRequest is temporarily on hold.
Active & Archived Requests Flow
flowchart TD A([Screen opens / resumes]) --> B["GET /users/{npi}/requests/active"] B --> C[Render active request cards] C --> D{User action} D -- Tap a request card --> E["Navigate to chat screen\nwith conversationId"] D -- Tap Archived --> F["GET /users/{npi}/requests/archived"] F --> G[Render archived request cards] G -- Tap a card --> E E --> H["Connect WSS /conversations/{id}/messages"] H --> I([Messaging flow — see Section 06])

Error Handling

HTTP Status Codes

CodeMeaning
200 OKRequest succeeded.
201 CreatedResource created successfully.
400 Bad RequestMissing or invalid parameters. See error body for details.
401 UnauthorizedAPI key is missing or invalid.
404 Not FoundThe requested resource does not exist.
409 ConflictResource already exists (e.g. duplicate user registration). The existing resource is returned.
429 Too Many RequestsRate limit exceeded. Retry after the duration in the Retry-After header.
500 Internal Server ErrorUnexpected error on DocUpdate's side. Contact your integration rep if this persists.

Error Response Format

JSON
{
  "error": {
    "code":    "MISSING_FIELD",
    "message": "The field 'npi' is required.",
    "field":   "npi"
  }
}

Retry Strategy

For transient errors (500, 503, network timeouts), implement exponential backoff:

AttemptWait Before Retry
1st retry500ms
2nd retry1,000ms
3rd retry2,000ms
After 3 retriesSurface error to user
Do not retry on 400 or 401 errors. These indicate a problem with the request itself that will not resolve on retry.
