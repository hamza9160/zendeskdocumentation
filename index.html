Partner Integration Guide

DocUpdate Concierge API

Concierge enables healthcare professionals to accelerate access to samples, patient support programs, education, reps, and more, all through a single integration with DocUpdate.

01

Overview

The DocUpdate Concierge API enables healthcare professionals (HCPs) to submit requests directly from your application. DocUpdate acts as the middleware layer between your application and our underlying messaging engine — your integration never communicates directly with any third-party messaging provider.

Capabilities

CapabilityDescription
User ManagementRegister and update HCP accounts (upsert, keyed by NPI)
Request SubmissionCreate a conversation by submitting a request type and message
MessagingRetrieve and send messages on a conversation via REST
Conversation TrackingRetrieve a user's active and archived conversations

Architecture

Your application communicates exclusively with DocUpdate endpoints. DocUpdate handles all downstream processing.

Your Application
HTTPS + API Key
DocUpdate Concierge API
api.docupdate.io/concierge/v1
Internal only — not exposed to partners
Messaging Engine

End-to-End Flow

A typical integration follows the sequence below — register the user on login, fetch the available request types, create a conversation, then exchange and track messages. Each step is detailed in the sections that follow.

End-to-End Integration Flow
sequenceDiagram participant App as Your Application participant API as DocUpdate Concierge API Note over App,API: On login (and whenever details change) App->>API: POST /users (upsert HCP by NPI) API-->>App: userId Note over App,API: Starting a request App->>API: GET /request-types API-->>App: request types (+ samplesList) App->>API: POST /conversations (requestType, message) API-->>App: conversationId Note over App,API: Messaging on a conversation App->>API: GET /conversations/{id}/messages API-->>App: messages + status App->>API: POST /conversations/{id}/messages (message) API-->>App: message Note over App,API: Viewing history App->>API: GET /users/{npi}/conversations/active API-->>App: active conversations

Messaging UI Ownership

DocUpdate provides REST endpoints for messaging. Your application is responsible for rendering the messaging interface. Messages are delivered with a role field (user or agent) that you can use to differentiate and style messages on each side of the conversation.

02

Authentication

All requests to the DocUpdate Concierge API are authenticated using an API key passed in the request header.

API Key Header

Header
X-DocUpdate-API-Key: your_api_key_here

Include this header on every request. Requests without a valid API key will receive a 401 Unauthorized response.

Rules

RuleDetail
TransportHTTPS only. HTTP requests will be rejected.
Key RotationContact your integration rep to rotate a compromised key.
EnvironmentsSeparate keys are issued for sandbox and production.
Obtaining an API Key: Contact your DocUpdate integration representative to receive your API key. Keep your key secure and do not expose it in client-side code.

04

User Management

Before an HCP can create conversations or send messages, they must be registered with DocUpdate. Registration should occur automatically following user login in your application, and again whenever the user's details change.

Register or Update User

POST/users

Upsert endpoint for HCP user accounts. The user is identified by NPI: if no account exists, a new one is created; if an account already exists, its details (email, name, phone) are updated with the values in the request. Call this endpoint both on first login and whenever the user's details change.

Request Body

JSON
{
  "npi": "1234567890",
  "email": "dr.smith@practice.com",
  "firstName": "Jane",
  "lastName": "Smith",
  "phone": "404-555-0100"
}
FieldTypeDescription
npirequired stringNational Provider Identifier. Used as the unique identifier for the user.
emailrequired stringUser's email address.
firstNamerequired stringUser's first name.
lastNamerequired stringUser's last name.
phonerequired stringUser's phone number.

Response — 201 Created / 200 OK

JSON
{
  "userId": "usr_abc123",
  "npi": "1234567890",
  "email": "dr.smith@practice.com",
  "firstName": "Jane",
  "lastName": "Smith",
  "phone": "404-555-0100",
  "createdAt": "2026-04-06T12:00:00Z"
}
User Registration Flow
flowchart TD A([App Login or details changed]) --> B["POST /users\nnpi · email · firstName · lastName · phone"] B --> C{HTTP Response} C -- 201 Created --> D[New user created] C -- 200 OK --> E[Existing user updated] D --> F([Store userId — ready to create conversations]) E --> F
05

Submitting a Request

Submitting a request is a two-call flow: first fetch the available request types (and, for sample requests, the list of available samples), then create a conversation using the selected type's id.

Get Request Types

GET/request-types

Returns the list of request types a user can choose from. Call this before creating a conversation and use it to populate the request-type selector. Each type's id is the value sent as requestType in Create Conversation below. The samplesList is present only for the samples request type and lists the medications the user can request.

Response — 200 OK

JSON
{
  "requestTypes": [
    {
      "id": 1,
      "requestType": "Request Free Samples",
      "samplesList": ["Ozempic", "Atorvastatin", "Mounjaro"]
    },
    {
      "id": 2,
      "requestType": "Schedule a Rep Visit"
    },
    {
      "id": 3,
      "requestType": "Custom Request"
    }
  ]
}
FieldTypeDescription
idintegerThe request type identifier. Send this as requestType when creating a conversation.
requestTypestringHuman-readable label for the request type. Use it as the selector option text.
samplesListoptional string[]List of available sample medications. Present only for the samples request type (id: 1).

Create Conversation

POST/conversations

Creates a new Concierge conversation for the specified user. The request type and initial message are submitted here, and a conversationId is returned for all follow-up messaging.

Request Body

JSON
{
  "requestType": 1,
  "message": "I need samples of Ozempic"
}
FieldTypeDescription
requestTyperequired integerThe id of the selected request type, obtained from GET /request-types above (1 = Request Free Samples, 2 = Schedule a Rep Visit, 3 = Custom Request).
messagerequired stringThe user's request message. Max 200 characters. For a samples request, include the requested medication names (from samplesList) in this message.

Response — 201 Created

JSON
{
  "conversationId": "conv_xyz789",
  "status": "new",
  "requestType": 1,
  "createdAt": "2026-04-06T12:05:00Z"
}
Store the conversationId returned. It is required to retrieve messages and send follow-up messages on this conversation.
Submit Request Flow
flowchart TD A([Open request form]) --> G["GET /request-types"] G --> H[Populate type selector + samples list] H --> I([User selects type + writes message]) I --> B["POST /conversations\nrequestType · message"] B --> C{HTTP Response} C -- 201 Created --> D[Store conversationId] C -- 4xx / 5xx --> E([Show error to user]) D --> F([Show success — request submitted])
06

Messaging

Messaging uses standard REST endpoints. Retrieve a conversation's message history with a GET request, and post new messages with a POST request. Poll the history endpoint to display new agent replies.

Get Messages

GET/conversations/{conversationId}/messages

Returns a page of messages for the specified conversation, along with the conversation's current status (see Section 07 — Status Reference). By default the most recent messages are returned; page backward through history using the cursor links below. Call this when opening the conversation, and poll periodically to fetch new agent replies and status changes.

Query Parameters

ParameterTypeDescription
page[size]optional integerNumber of messages per page. Fixed at 100 for the messages endpoint.
page[before]optional stringCursor pointing to a message id. Returns messages older than (preceding) the cursor — used to load earlier history. Prefer following the pagination.prev URL.
page[after]optional stringCursor pointing to a message id. Returns messages newer than (following) the cursor. Prefer following the pagination.next URL.
Pagination is cursor-based, mirroring Sunshine Conversations. With no cursor, the most recent page is returned (newest history first). To load older messages, follow the pagination.prev URL (or pass page[before]); to fetch newer messages, follow pagination.next (or page[after]). The pagination.hasMore flag indicates whether more messages exist in the direction you are paging. Page size is fixed at 100.

Response — 200 OK

JSON
{
  "conversationId": "conv_xyz789",
  "status": "open",
  "messages": [
    {
      "messageId": "msg_041",
      "role": "user",
      "text": "Any update on my sample request?",
      "sentAt": "2026-04-08T09:15:00Z"
    },
    {
      "messageId": "msg_042",
      "role": "agent",
      "text": "Your samples have shipped — tracking to follow.",
      "sentAt": "2026-04-08T09:20:00Z"
    },
    {
      "messageId": "msg_043",
      "role": "user",
      "text": "Great, thank you!",
      "sentAt": "2026-04-08T09:22:00Z"
    }
  ],
  "pagination": {
    "hasMore": true,
    "prev": "https://api.docupdate.io/concierge/v1/conversations/conv_xyz789/messages?page[before]=msg_041",
    "next": null
  }
}

Send Message

POST/conversations/{conversationId}/messages

Posts a new message from the HCP on the conversation.

Request Body

JSON
{
  "message": "Thanks! Can you share the tracking number?"
}
FieldTypeDescription
messagerequired stringThe message text to post on the conversation. Max 200 characters.

Response — 201 Created

JSON
{
  "messageId": "msg_044",
  "role": "user",
  "text": "Thanks! Can you share the tracking number?",
  "sentAt": "2026-04-08T09:25:00Z"
}
UI Tip: Use the role field to position messages in your interface. Messages with role: "user" should appear on the right; role: "agent" on the left.
Messaging Flow
flowchart TD A([Open conversation screen]) --> B["GET /conversations/{id}/messages"] B --> C[Render message history] C --> D{User action} D -- Poll for updates --> E["GET /conversations/{id}/messages"] E --> F[Append new agent replies] F --> D D -- User sends message --> G["POST /conversations/{id}/messages\nmessage"] G --> H[Append sent message to UI] H --> D
07

Conversation Management

Use these endpoints to retrieve a user's conversation history. Each item in the response includes a conversationId — this is the same ID used to retrieve and send messages for that conversation (see Section 06 — Messaging).

Get Active Conversations

GET/users/{npi}/conversations/active

Returns active conversations for the specified HCP (paginated). Active conversations have a status of new, open, or pending.

Query Parameters

ParameterTypeDescription
perPageoptional integerNumber of conversations per page. Default 100, maximum 100.
Pagination is offset-based (mirroring the underlying ticketing system). Omit query parameters on the first request to get the first page; then follow the pagination.nextPage URL from each response to fetch subsequent pages. When nextPage is null, you have reached the last page. The maximum perPage is 100; larger values are capped at 100.

Response — 200 OK

JSON
{
  "conversations": [
    {
      "conversationId": "conv_xyz789",
      "requestType": 1,
      "status": "open",
      "latestMessage": "Your samples have been shipped.",
      "hasNewResponse": true,
      "createdAt": "2026-04-06T12:05:00Z",
      "updatedAt": "2026-04-06T14:00:00Z"
    }
  ],
  "pagination": {
    "perPage": 100,
    "count": 135,
    "nextPage": "https://api.docupdate.io/concierge/v1/users/1234567890/conversations/active?perPage=100&page=2",
    "previousPage": null
  }
}
FieldTypeDescription
conversationIdstringUnique conversation identifier. Use it to retrieve and send messages (see Section 06).
requestTypeintegerThe request category the conversation was created with (see Section 05).
statusstringCurrent conversation status (see Status Reference below).
latestMessagestringPreview text of the most recent message on the conversation. Use it to render a snippet in the conversation list.
hasNewResponsebooleantrue when there is an unread agent reply. Use it to show an unread badge; it is cleared once the user opens the conversation.
createdAtstringISO 8601 timestamp of when the conversation was created.
updatedAtstringISO 8601 timestamp of the last activity on the conversation.

Get Archived Conversations

GET/users/{npi}/conversations/archived

Returns archived conversations (paginated). Archived conversations have a status of solved or closed. Accepts the same perPage query parameter and returns the same response shape (including the offset-based pagination object) as Get Active Conversations above.

Status Reference

StatusCategoryDescription
newActiveConversation received, awaiting initial response.
openActiveConversation is being actively worked on.
pendingActiveAwaiting a response from the HCP.
solvedArchivedConversation was completed successfully.
closedArchivedConversation was resolved and closed.
Active & Archived Conversations Flow
flowchart TD A([Screen opens / resumes]) --> B["GET /users/{npi}/conversations/active"] B --> C[Render active conversation cards] C --> D{User action} D -- Tap a conversation card --> E["Navigate to chat screen\nwith conversationId"] D -- Tap Archived --> F["GET /users/{npi}/conversations/archived"] F --> G[Render archived conversation cards] G -- Tap a card --> E E --> H["GET /conversations/{id}/messages"] H --> I([Messaging flow — see Section 06])

Error Handling

HTTP Status Codes

CodeMeaning
200 OKRequest succeeded.
201 CreatedResource created successfully.
400 Bad RequestMissing or invalid parameters. See error body for details.
401 UnauthorizedAPI key is missing or invalid.
404 Not FoundThe requested resource does not exist.
409 ConflictThe request conflicts with the current state of the resource.
500 Internal Server ErrorUnexpected error on DocUpdate's side. Contact your integration rep if this persists.

Error Response Format

JSON
{
  "error": {
    "code": "MISSING_FIELD",
    "message": "The field 'npi' is required.",
    "field": "npi"
  }
}

The error.code is a stable, machine-readable string — branch on it in your integration rather than parsing error.message, which is for humans and may change.

Error Codes

CodeHTTPDescription
MISSING_FIELD400A required field is absent. error.field names it.
INVALID_FIELD400A field is present but malformed (e.g. NPI not 10 digits). error.field names it.
INVALID_REQUEST_TYPE400The requestType id is not one returned by GET /request-types.
UNAUTHORIZED401The API key is missing or invalid.
NOT_FOUND404The requested user or conversation does not exist.
CONFLICT409The request conflicts with the current state of the resource.
INTERNAL_ERROR500An unexpected error occurred on DocUpdate's side.
New error codes may be introduced over time. Treat any unrecognized error.code as a generic failure for its HTTP status, rather than failing hard.
